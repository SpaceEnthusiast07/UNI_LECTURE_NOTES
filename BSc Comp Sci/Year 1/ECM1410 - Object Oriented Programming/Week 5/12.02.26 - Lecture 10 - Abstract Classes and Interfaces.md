##### Reference vs. Instantiation
 - For binding, it is important to distinguish these two concepts.
 - Say you have the code: `Animal myPet = new Dog();`
	 - The variable `myPet` has a *reference type* `Animal`, but an *instantiated/object type* of `Dog`.
	 - Therefore, if a method is called, e.g. `myPet.bark();`, that isn't in the `Animal` class, the compiler will throw a `Symbol not found` exception because it only looks in the reference type - the `Animal` class, not the `Dog` class.
 - However, if you define an abstract method in the `Animal` class, then the Java Runtime Environment (JVM) will use the method body or implementation defined in the instantiation/object type - the `Dog` class.


>[!Warning] Overridden Methods are also known as Virtual Methods.


##### Binding
 - Binding is the process of linking a method call to the specific version of that method.
 - The whole reason why Polymorphism works is because of binding.
 - There are two types of binding:
	 - **Static Binding**
		 - This is where method calls and method bodies are linked at compile time.
		 - Only methods that cannot be overridden, such as `static`, `private`, or `final`, are statically linked.
		 - This is because, if they were overridden, the compiler will not know which version of the method to link, as there are several with the exact same method signature.
		 - Since the method could be overloaded within the same class, the compiler will use the parameters passed in to determine the exact overloaded method to link.
		 - To determine which class to look within to find the method, the compiler looks at the reference type of the object, as explained in the above notes. The compiler only looks within this class and no subclasses.
	 - **Dynamic Binding**
		 - This method of binding is performed on overridden methods (virtual methods).
		 - Say you defined a variable with the reference type `Shape` and instantiated it with a `Circle` object, and you call `.draw()`, the compiler doesn't know which method to call as the object could be a `Triangle` or `Rectangle`.
		 - Therefore, during runtime, the JVM looks at the actual object in memory, then decides which version of `.draw()` to call.
	 - Finally, if a method is `static`, `private`, or `final`, any method calls are static binding, otherwise, if the method has the possibility of being overridden, dynamic binding is used!

>[!Warning] Fields (attributes) are always statically bound to the reference type.


##### Polymorphism
 - The mechanism of performing a single action in different ways.
 - Polymorphism is implemented through method overriding.
 - Example:
```java
public class Animal {
	// Private attributes
	// ...
	
	// Constructor
	// ...
	
	public void makeNoise() {
		// Do something generic
		System.out.println("Boo!"); 🤣
	}
	
	// Other methods
	// ...
}

public class Dog extends Animal {
	// Private attributes
	// ...
	
	// Constructor
	// ...
	
	public void makeNoise() {
		// Do something dog-like
		System.out.prinln("Woof!");
	}
	
	// Other methods
	// ...
}

public class Cat extends Animal {
	// Private attributes
	// ...
	
	// Constructor
	// ...
	
	public void makeNoise() {
		// Do something cat-like
		System.out.prinln("Meow!");
	}
	
	// Other methods
	// ...
}
```


##### Abstract Classes
 - As you may have noticed in the above example, if someone calls the `makeNoise()` method from the `Animal` class, what noise do you make?
 - This is where `abstract` classes come in.
 - Abstract classes cannot be instantiated directly, only their child classes can be.
 - Then, within this class, you can define any methods you want, but you don't provide any instructions within the method body.
 - This is useful in the scenario above, where the `makeNoise()` method in the `Animal` class doesn't make sense logically, so `Animal` can be made abstract and the `makeNoise()` method can also be made abstract.
 - An **abstract method** is one *without an implementation*.
 - An abstract class may not have any abstract methods.
 - Example:
```java
// Abstract Class
public abstract class Animal {
	// ...
	
	// Format of an abstract method
	public abstract void makeNoise();
}

// Regular child class
public class Dog extends Animal {
	// ...
	
	// Now, define the method for the type Dog
	public void makeNoise() {
		System.out.println("Woof!");
	}
}
```

>[! Warning] Calling something like `Animal.makeNoise();` will result in a compiler error!


##### Abstract Rules
 - If a class contains an abstract method, it must be declared as abstract.
 - If a class extends an abstract class, then:
	 - All abstract methods must be defined in the subclass.
	 - Or, the subclass is defined as abstract itself.


##### Abstraction
 - **Abstraction** is the concept of *hiding/removing all but relevant data* in order to *reduce complexity* and *increase efficiency*.
 - Encapsulation is a kind of abstraction.
 - Abstract classes achieve partial abstraction.
 - Interfaces achieve full abstraction (almost). 🤣


##### Interfaces
 - It is sometimes useful to *define* what a class *must do*, but *not how to do it*.
 - That is what an interface is, a fully abstract class, containing only abstract methods.
 - Therefore, when a class *implements* an interface, it must provide its own implementation of the methods defined in the interface.
 - This diagram shows which keyword is used depending on the context: ![[Pasted image 20260212152648.png]]
 - A single class can only extend on class at a time, but it can implement more than one interface at a time.


##### Defining an Interface
 - It has *no constructors*.
 - It only contains public, static and final attributes.
 - It can only contain public and abstract methods.
 - Since Java 8, you can have static and default methods.


##### Default Methods
 - When a parent type (class or interface) defines a default method, one of three things happen:
	 1. Override the default method like normal.
	 2. Declare the method again with no body, i.e. making it abstract.
	 3. Or, no mention means it is simply inherited as usual.


##### Abstract Classes vs. Interfaces
 - Similarities:
	 - Both cannot be directly instantiated.
	 - Both can include abstract methods.
	 - Both are usually used to define what you want doing, but leave the actual implementation to the subclasses.
 - Differences:
	 - Constructors ... as mentioned above
	 - Attributes ... as mentioned above
	 - Methods ... as mentioned above


##### Rule of Thumb
 - Always use interfaces.
 - "Program to an interface, not an implementation."
 - The separation of interface and implementation enables better software design and ease in expansion.
