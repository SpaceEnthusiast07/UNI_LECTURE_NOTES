##### What are States?
 - Logically, you have two main types of states:
	 - **Passive** (inactive)
	 - **Active** (doing something)
 - A **state** represents the history of inputs/events on an object.
 - A change of state may result in an output of information.
 - State machine diagrams can be heavily used in some parts of a software system.
 - E.g. safety critical systems, vending machines, computer games, etc.


##### Overview of State Machine Diagrams
 - States are represented by rounded rectangles: ![[Pasted image 20260319144653.png]]
 - A **transitions** are *legal moves between states* and are *denoted by an arrow* between two states.
 - The external event causing the change of state (i.e. a trigger) is written above the transition arrow.
 - A change of state can also be caused by an internal trigger.
 - Explaining the above example: ![[Pasted image 20260319145024.png]]
	 - Some object, e.g. a box, has two states "Opened" and "Closed".
	 - To transition from state "Opened" to "Closed", you must "lower the lid".
	 - Then, you must "open the lid" to transition from "Closed" to "Opened".

##### Conditional Transitions
 - A state machine can have more complicated transitions than what is shown in the above examples.
 - A **guard** is a *Boolean condition*, which *if true, permits the transition*, otherwise the state is unchanged.
 - Guard structure:
	 - `trigger[guard]/behaviour`
	 - Each element is optional
 - Once a transition has started, you cannot interrupt it.
 - Example: ![[Pasted image 20260319150516.png]]
	 - I.e. if the keystroke is not the Enter key, wait for another input.
	 - Then, if the keystroke is the key "Enter", by pressing "Enter" you are "Submitting input" and transition to the "Processing Input" state.

 - If neither a trigger nor a guard is specified on a transition, this means that once the internal behaviour has completed, this transition is automatically taken.

##### Behaviours
 - "Do behaviour" is behaviour that happens as long as the state is active.
 - I.e. repeats endlessly unless an external trigger forces it to end.
 - This type of behaviour may complete on its own, or it is forced to complete by a trigger that causes a state change.


##### State Machines for Software
 - They model the life cycle of an object.
 - These diagrams are only worth the effort for complicated objects.
 - They are useful for modelling object behaviours which change depending on state.
 - Within code, triggers are method calls.


##### Types of Behaviours
 - Different things (behaviours) can happen when entering, inside, and exiting a state.
 - These behaviours have certain keywords:
	 - `entry/behaviour` - when entering the state, do `behaviour`.
	 - `do/behaviour` - do endlessly until an external trigger causes a state change.
	 - `end/behaviour` - when exiting a state, do `behaviour`.
 - Example: ![[Pasted image 20260319163009.png]]


##### Internal Transitions
 - These cause a reaction within the state, but do not cause a state change.
 - Such an internal transition is when transitioning from the `entry` to `do` behaviour in the example above.
 - These are separate from self transitions.


##### Composite States
 - These allow for parallel or simultaneous states within an object.
 - Example: a person (object) is walking (state 1) and talking (state 2) at the same time.

##### Summary of State Machines
 - Can be used to model systems, components and objects.
 - State machines are very useful for describing objects, the various states they can be in, and the causes for state change.
 - They are excessive for simple objects, but help to remove ambiguities in systems design.
