##### UML Object Diagrams
 - They are very similar to class diagrams, but simpler.
 - Effectively an *instance of a class diagram* - shows a snapshot of the state of the system at a particular point in time.
 - Useful when reasoning through particular scenarios of complex object interactions.
 - As with class diagrams, objects are represented using rectangles, but the name is underlined.

##### UML Sequence Diagrams
 - A sequence diagram is a type of UML interaction diagram.
 - "Interaction diagram" means it shows how objects communicate and interact with each other.
 - Concerned with the *order* of interactions between parts of a system.
 - Participants within a sequence diagram are the interactive parts of the system.
 - Structure:
	 - Each participant is organised horizontally and not overlapping.
	 - Each have a plum line below them which is there "lifeline".
	 - Participants are usually software objects, but can also be other parts of the system.
 - Example structure: ![[Pasted image 20260317111415.png]]
 - Participant name structure: ![[Pasted image 20260318131358.png]]


##### Time in UML Sequence Diagrams
 - Time starts at the top of the page and progresses down the page.
 - Therefore, the vertical order of operations is their order of execution in time.
 - These diagrams are concerned with the order of operations, not their duration.
 - That means, the gaps between operations is not indicative of the exact time between them, it only indicates that there is a portion of time where nothing happens.


##### Representing Interactions
 - Communications between participants is denoted by *activation bars* and *message arrows*.
 - An **activation bar** is simply a solid white rectangle showing that this participant is active at this time.
 - The length of these bars is not indicative of the actual time spent active, just a guide.
 - Example: 
   ![[Pasted image 20260318131834.png]]
 - Just because the second bar is half the size doesn't mean it is active for half the time, it just means the the second one finishes before the first.
 - The **message arrow** in the above diagram shows that participant 1 activates participant 2.

##### Types of Message Arrows
 - You can have different types of message arrows used for different types of interactions.
 - **Synchronous message**
	 - This is where the sender waits for the receiver to complete its tasks before the sender can continue.
	 - This interaction is denoted by an arrow with a solid black head.
	 - E.g. the `poke()` interaction in the above example.
 - **Asynchronous message**
	 - This is where the sender does not wait for the receiver to complete its tasks.
	 - This type is denoted by an arrow head this is empty: ![[Pasted image 20260318133017.png]]
 - Return message
	 - This arrow is optional, but explicitly shows the return of control from one participant to another.
	 - Denoted by a dashed arrow.
 - You also have participant creation and destruction message arrows.
