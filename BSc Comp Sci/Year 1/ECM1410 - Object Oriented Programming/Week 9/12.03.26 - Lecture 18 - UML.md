##### Unified Modelling Language (UML)
 - UML is the standard for modelling software systems.
 - Properties:
	 - Formal - Elements are strongly defined.
	 - Concise - Straightforward and simple notation.
	 - Comprehensive - Covers all important aspects of a system.


##### UML 2.0
 - The first UML version covered designs and requirements of software exclusively.
 - UML 2.0 allows you to model other systems other than just software.


##### Ways to Use UML
 - **Sketch** - convey key points in development.
 - **Blueprint** - detailed specification of a system.
 - **Programming language** - tools convert UML to executable code.


##### General Software Development
 - Waterfall process
	 - Set requirements early
	 - Design in full then implement
	 - Eggs in the basket
 - Iterative process
	 - Multiple development phases
 - Agile methods
	 - Minimise risk when working with a system of expending capabilities.
	 - Rapid iterations
 - UML is used in all the above.


##### Kruchten’s 4+1 View Model
 - A framework designed to describe software architecture using five concurrent, stakeholder specific views.
 - Logical - System parts; class, objects, state machine and interaction diagrams.
 - Process - Activity diagrams.
 - Development - Package and component diagrams.
 - Physical - Deployment diagrams.
 - Use case - Use case and overview diagrams.


##### UML Diagrams
 - Class diagrams
 - Object diagrams
 - Package diagrams
 - Sequence diagrams
 - State machine diagrams
 - And more diagrams

##### Properties and Arrays
 - Properties provide extra information for an attribute or method.
 - They are specified within curly braces `{}`.
 - E.g. `# birthday : Date {readOnly}`.
 - Arrays are denoted with square brackets `[]`.
 - Within the square brackets, you specify the bounds of the number of elements within this array.
 - E.g. `~ names : String[*] {unique}`.
 - Here, the `*` means that the array can take any number of elements.
 - Also, `{unique}` means that the array doesn't accept duplicate elements.


##### Multiplicity
 - The use of arrays/collections of objects.
 - Can be shown both for inline attributes and attributes by association.

##### Relationships between Classes
 - The different types of relationships are listed in order of increasing relationship strength:
	 - Dependency
	 - Association
	 - Aggregation
	 - Composition
	 - Inheritance


##### Dependency
 - Denoting that one class has a need to know about another to make use of it.
 - This is the weakest relationship between objects.
 - E.g. class B may have a method which briefly uses class A, however this is local to a method.
 - The symbol used is a dashed line, with an arrow pointing to the class that is used.
 - Example:
![[Pasted image 20260312173430.png]]
 - The `Game` class may use the `Random` class once to generate a random number.

 - Denotes that one class will work with one or more instances of another class for a prolonged period of time.

##### Aggregation
 - A stronger from of association.
 - Indicates that one class owns references to objects of another class (though may also share them).
 - This "owning" is logical, i think...
 - The symbol used is a solid line with an empty diamond next to the "owning" class.
 - E.g. an `Author` object will "own" multiple `Book` objects.
 - Example:
![[Pasted image 20260312174318.png]]


##### Composition
 - A stronger form of aggregation.
 - If instances of class B and C have a composition relation ship with class A, they are actually integral parts of A.
 - I.e., class a cannot function without class B and C.
 - Class B and C are not usually used by any other part of the system.
 - The symbol used is a filled in diamond next to the owning class.
 - Example:
![[Pasted image 20260312174446.png]]


##### Inheritance
 - Inheritance is denoted by an empty triangle next to the parent/super class.
 - When subtyping a class, a solid line is used.
 - However, when subtyping an interface, a dashed line is used, referred to as a realisation.
