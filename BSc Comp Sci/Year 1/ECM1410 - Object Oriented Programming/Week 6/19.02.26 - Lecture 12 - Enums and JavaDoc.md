#### Enums
##### What is an Enum?
 - An `enum` type/class is a special reference type that holds a set of predefined constants.
 - Example definition structure in Java:
```java
enum Day {
	MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY, SATURDAY, SUNDAY
}
```
 - By convention, all elements in the `enum` are capitalised.
 - All constants in an `enum` are defined implicitly as `public`, `static`, and `final`.


##### Using Enums
 - You do not instantiate an `enum` class/type.
 - They are pre-allocated and fixed.
 - Example: `Day d = Day.SUNDAY;`.
 - To get all the values of an `enum` use: `Day[] allDays = Day.values();`.
	 - Each `enum` type has a static method `values()`.
 - Example Java program from slides:
```java
// A demo to comment according to which day today is
enum Day { // The values are separated by comma
	SUNDAY, MONDAY, TUESDAY, WEDNESDAY,
	THURSDAY, FRIDAY, SATURDAY
}

public class EnumApp {
	public static void main(String[] args) {
		Day today = Day.FRIDAY;
		// Output the day
		System.out.println("Today is "+today+"\n");
		for (Day d:Day.values()) // List all seven days
		giveComments(d);
	}

	public static void giveComments(Day day) {
		switch (day) {
		case MONDAY:
		System.out.println(day+": Mondays are bad.");
		break;
		case FRIDAY:
		System.out.println(day+": Fridays are better.");
		break;
		case SATURDAY: case SUNDAY:
		System.out.println(day+": Weekends are best.");
		break;
		default:
		System.out.println(day+": Midweek days are so-so.");
		}
	}
}
```

#### Javadoc
##### Java Documentation
 - Comments are used to explain code and aid maintainability.
 - Single line comment: `// This is a single line comment`.
 - Multi-line comment:
```java
/*
This is a
multi-line comment
*/
```
 - Documentation comment:
```java
/**
This is a Javadoc comment
that describes a class or method.
*/
```


##### What is Javadoc?
 - **Javadoc** is a *JDK tool* that *automatically creates* fancy *HTML-based documentation* based on the documentation comments (`/** ... */`).
 - Example output: ![[Pasted image 20260219151252.png]]
 - Before each class and/or method, you can provide a "doc string" (documentation comment) that gives a description for that class and/or method. ![[Pasted image 20260219151341.png]]


##### Using Javadoc
 - Many IDEs have a simple button that you can click to generate Javadoc for your whole project.
 - You can also simply create Javadoc for a specific `.java` file: `$ javadoc myProgram.java`.
 - To create Javadoc for all java files in the current directory, type: `$ javadoc *.java`.


##### Doc Tags
 - Within the doc string, you can specify special **doc tags** that *provide specific information* about what you are describing and are *used to further enhance the HTML documentation*.
 - These include:
	 - `@author` - Provides information about the author.
	 - `@version` - Indicates the version number.
	 - `@since` - The version this class/method was added in.
	 - `@param` - Specified for each parameter, defining its name and purpose.
	 - `@return` - More information about what this function returns, if anything.
	 - `@throws` - For each exception thrown, this tag provides information about why this exception might be thrown.


##### Annotations
 - These are like doc tags in the sense that they start with a `@` followed by the name of the annotation.
 - Unlike doc tags which are only check by Javadoc, annotation are checked by the compiler.
 - 3 most common annotations:
	 - `@Deprecated` - implies that this method or class is no-longer recommended or supported. Used alongside the Javadoc tag `@deprecated`.
	 - `@Override` - indicates that this method overrides the method defined in the parent class. An error is thrown when this method doesn't override anything.
	 - `@SuppressWarnings("warning")` - informs the compiler that the dev is aware of the error and to ignore it (do not output the corresponding error message).
