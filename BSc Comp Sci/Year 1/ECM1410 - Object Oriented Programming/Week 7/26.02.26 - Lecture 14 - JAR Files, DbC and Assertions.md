#### JAR Files
##### What is a JAR File?
 - The Java archive file (`.jar`) is a compressed archive that the `jar` utility uses to store all the resources needed to run the application.
 - It has a standard structure allowing you to easily share applications/packages/libraries.
 - The `.jar` file contains all `.class` files along with images and other data.
 - It may also contain `.java` files, but rarely.
 - You are able to configure the `.jar` file to run like a regular application by double clicking it.
	 - To achieve this, a `MANIFEST.MF` file needs to be present in the `.jar`, which is used to know which `.class` file is the main entry point.


##### JAR CLI Command
 - You can use the CLI command `jar` to create JAR files.
 - Structure: `jar [options] [jar_name] [directory_of_class_files]`
 - Example: `jar -cvfe ./build/myApp.jar MainEntryPoint -C bin .`
	 - `-c` - used to create a new JAR file.
	 - `-v` - verbose output, i.e. "tell me what you add to the JAR file".
	 - `-f` - next argument is the file name of the JAR to be created: `./build/myApp.jar`.
	 - `-e` - the name of the class which is the entry point to the application: `MainEntryPoint`.
	 - `-C bin .` - tells `jar` to go INTO the bin directory and package everything up in there.
 - The `-C` flag must be present as all the Java package folders need to be in the top level of the `.jar` file.
	 - If you didn't, the JAR file would look like: `myApp.jar/bin/com/myapp/Main.class`.
	 - But with it: `myApp.jar/com/myapp/Main.class`.
 - You can list the contents of a JAR file by using the following command:
	 - `jar -tvf myJarFile`
	 - The `v` and `f` have the same meaning as in the command above, however the `t` stands for `table of contents`.
 - You are also able to extract the files within a JAR file using:
	 - `jar -xvf myJarFile`
	 - Where the `x` stands for `extract`.


##### The Manifest File
 - This file is named `MANIFEST.MF` and lives in the directory `META-INF` inside the `.jar` archive.
 - The Manifest is the instruction manual for the JAR file and informs the Java Runtime (JVM) how to use the files within the archive.
 - Without this file, a JAR archive is simply a fancy zip file.
 - It contains data in the form of `key : value`.
 - It has 3 main jobs:
	 - *Defining the Entry Point*
		 - This value tells the JVM, "When someone double-clicks this JAR, run the `main` method within this specific class."
		 - Example: `Main-Class: com.myapp.MainEntryPoint`.
	 - *Managing Dependencies*
		 - If your app needs other JAR libraries to function, the Manifest lists where to find them.
		 - E.g. `Class-Path: lib/helper.jar`.
	 - *Version Control*
		 - It also stores version numbers, who created the file, and the build dates.

>[!Warning] A MANIFEST.MF file ALWAYS ends with a newline!

#### Design by Contract
##### What is DbC
 - A programming style in which the invocation of a method is viewed as a contract between a client and server, with each having *explicitly stated responsibilities*.
 - **Pre-condition**: What must be true before the method is called.
	 - The method tells the client: "this is what I expect from you".
 - **Post-condition**: What must be true after the method completes successfully.
	 - The method tells the client: "this is what I promise to do".
 - To implement this methodology, a method must check the validity of all pre-conditions, and raise the respective exception if one is false.
 - Therefore, the method shouldn't start any of its functionality/processing until it knows all the pre-conditions are met.


##### Class Invariants
 - **Class invariants** refer to rules that must be true about all class instances and before/after ANY public method is called, and even the constructor.
 - *Example*: Imagine the class `SavingsAccount` where the "law" is that the balance can never be negative.
	 - The class invariant would be: $\text{balance} \geq 0$.
	 - The constructor must ensure the initial deposit makes the balance $\geq 0$.
	 - With the `withdraw(amount)` method:
		 - *Pre-condition*: $\text{amount} > 0$.
		 - *The invariant's role*: The method must check that $\text{balance} - \text{amount} \geq 0$, if it isn't, the method would throw an exception.
		 - *Post-condition*: `balance` is decreased by `amount`.

>[!Warning] A public method is allowed to temporarily break the class invariant rule as long as it fixes the issue before it finishes.

#### Assertions
##### What are Assertions?
 - **Assertions** are used to verify assumptions that the programmer believes to be correct.
 - Structure:
	 - `assert <boolean_expr>` - if the expression is `false`, the JVM will throw an `AssertionError`.
	 - `assert <boolean_expr> : <message>` - if the expression is `false`, the JVM will throw an `AssertionError` along with the provided message.
 - Example:
	 - `assert Double.compare(b*b, a) == 0.0 : b + " is not the square root of " + a;"`.
	 - If `b` is not the square root of `a`, an `AssertionArrow` is raised.
 - Assertions are *disabled by default*, as they are used to test for fatal errors in design and therefore, shouldn't be shipped with the final release.
 - To use assertions:
	 - First compile as usual: `javac AssertionApp.java`
	 - Add the flag `-ea` or `-enableassertions` when executing the program:
		 - `java -ea AssertionApp`


##### Where should you use Assertions?
 - They are useful in detecting bugs and can be used for verifying:
	 - *Internal invariants*
		 - Assert that a value is within a certain constraint.
		 - E.g. `assert (x < upper) && (x > lower);`.
	 - *Control-flow invariants*
		 - Assert that a certain location will not be reached.
		 - E.g. `default: assert false; // inside switch-case block`.
	 - *Class invariants*
		 - Assert that an object's state is within a constraint.
		 - E.g. `assert isValid();`.
	 - *Pre-conditions* - for non-public methods.
	 - *Post-conditions* - ensure a method complies with its promise.


##### Where should you NOT use Assertions?
 - You should never use assertions on the pre-conditions of PUBLIC methods!
 - These should always be delt with using specific exceptions.
 - Also, they should never be used to implement application functionality, example what is not allowed: `assert age++ > 12;`.


##### Assertions vs. Exceptions
 - *Assertions*
	 - Used for internal logic checks within your code.
	 - I.e., making a program **correct**
 - *Exceptions*
	 - Used for "exceptional" conditions outside your immediate conde's control.
	 - E.g. invalid user input, missing files, etc.
	 - I.e., making a robust program.
