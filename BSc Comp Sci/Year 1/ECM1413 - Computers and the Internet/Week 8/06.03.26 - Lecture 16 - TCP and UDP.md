##### What is the Transport Layer?
 - The protocols involved are **TCP** and **UDP**.
 - The transport layer is responsible for the *end-to-end* transfer of application data.
 - **End-to-end** - from source to destination, not individual devices in the path to destination.
 - The transport layer is end-to-end meaning that once the data has arrived at the correct LAN/router, this layer directs the data to the specific device on the LAN/connected to the router.
 - Going down the network layer stack, it prepares application data for transport over the network.
 - Then, going up the layer stack, it processes network data ready for applications to use.
 - Household Analogy: This is used to distinguish between the transport and network layers.
	 - Hosts = houses
	 - Processes = the kids living in the houses
	 - Application Messages = the letters the kids are writing
	 - Transport Protocol = the person in the house that sorts the mail from the front door to a specific sibling (demultiplexing).
	 - Network-layer Protocol = the postal service that moves letters between houses.


##### TCP vs. UDP
 - Application developers choose a protocol based on their specific needs.

| Feature     | TCP (Transmission Control Protocol)                                  | UDP (User Datagram Protocol)                                   |
| ----------- | -------------------------------------------------------------------- | -------------------------------------------------------------- |
| Reliability | Reliable, since it acknowledges and resends lost data                | Does not resend lost data                                      |
| Ordering    | Uses sequence numbers to reorder the data if it arrives out-of-order | Data is processed as it arrives, it may be out of order        |
| Connection  | Connection oriented                                                  | Connectionless, i.e. send data even if destination is offline. |
| Overhead    | Higher overhead and slower                                           | Low overhead and fast                                          |
| Use Cases   | HTTP, Email (SMTP/POP), FTP                                          | VoIP, streaming video, DNS, online games, DHCP                 |

>[!Tip] HTTP/3 uses QUIC (based on UDP) to achieve both reliability and speed. 


##### TCP and UDP Shared Functions
 - **Segmentation**
	 - Large application data is split into smaller, manageable units.
 - **Multiplexing**
	 - Allows multiple applications (web, email, IM) to share the same network connection simultaneously.
 - **Encapsulation**
	 - Adding a header to the segmented data to create a *TCP segment* or *UDP datagram*.
 - **Addressing**
	 - *Well-known Ports (0-1023)*: Used for common server applications like HTTP (80) or SMTP (25).
	 - *Registered Ports (1024-49151)*: For specific user processes like MySQL (3306).
	 - *Dynamic Ports (49152-65535)*: Assigned to client applications when they start a connection.
 - **The Socket**
	 - A unique connection is defined by the combination of an IP address and a Port Number (e.g. `198.16.23.213:80`).
	 - Format: `IP:Port`.


##### Header Analysis
 - **UDP Header** (8 bytes) in includes:
	 - Source port (16 bits)
	 - Destination port (16 bits)
	 - Length (16 bits)
	 - Checksum (for error detection, 16 bits)
 - **TCP Header** (20 bytes) includes these extra fields:
	 - *Sequence number (32 bits)*: Tracks the position of data in the byte stream.
	 - *Acknowledgement Number (32 bits)*: Indicates the next expected byte.
	 - *Window Size (16 bits)*: Controls the amount of data a source can send before waiting for an ACK (Acknowledgement response).
	 - Code Bits: Control bits used during handshakes (e.g. SYN, ACK, FIN).


#### Advanced TCP Functions
 - TCP performs additional complex tasks that UDP does not:
##### Segment Reassembly
 - TCP uses In-Order Reassembly, whereas UDP uses Out-of-Order Reassembly.
 - Since segments can take different routes through the internet for a variety of reasons, such as congestion on one route, they may arrive out of order.
 - In this case, since TCP uses sequence numbers, it can reorganise the segments back into the correct order.
 - Therefore, since UDP doesn't use sequence numbers, if the data arrives out of order, the application will also receive it out of order, i.e. UDP cannot reorder segments.
 - This is one of the over heads TCP provides, which makes it slower, but more reliable. ![[Pasted image 20260408122719.png]]

##### TCP Connection Establishment
 - Establishing a connection uses 3 steps (a 3-way handshake): SYN, SYN-ACK, ACK.
 - Process:
	 - The host initiating the communication (host A) will send a TCP segment with sequence number 100 and the control to SYN (synchronise), to host B.
	 - The receiver (host B) then sends a SYN.ACK (synchronise-acknowledgement) back, with sequence number 300, control set to SYN.ACK, and the acknowledgement number to 101, meaning that the receiver (host B) is expecting segment with sequence number 101 next.
	 - Then, host A, sends back an ACK (acknowledgement) with sequence number 101, control set to ACK, and acknowledgement set to 301, meaning that host A expects a segment with sequence number 301 next, from host B.

##### TCP Connection Termination
 - Terminates the communication using 4 steps (a 4-way handshake): FIN, ACK FIN, ACK.
 - Process:
	 - Host A sends a FIN (finalise) TCP segment to host B.
	 - Host B then sends an ACK (acknowledgement) segment to A as well as a FIN segment.
		 - This step is sometimes combined into 1 in textbooks.
	 - Host A then sends an ACK back to B.
	 - Now the communication is terminated.

##### Acknowledgment of TCP Segments
 - Sending an acknowledgment every time you receive a segment adds too much overhead and slows the transfer down. ![[Pasted image 20260408124519.png]]
 - This is why we use **window size**, which is the amount of data (bytes) a sender can send before waiting for an acknowledgement.
 - If the acknowledgement is not received in time, the sender retransmits the missing data, i.e. all the data since the last acknowledgement. ![[Pasted image 20260408124443.png]]


##### TCP Congestion and Flow Control
 - **Flow control** is where the receiver can *adjust the window size* to inform the sender of how much data it can receive, *preventing the receiver from becoming overwhelmed*.
 - **Congestion Control**: If segments are lost due to network congestion, the receiver acknowledges the last successful segment and reduces its window size to slow down the sender. ![[Pasted image 20260408124826.png]]
