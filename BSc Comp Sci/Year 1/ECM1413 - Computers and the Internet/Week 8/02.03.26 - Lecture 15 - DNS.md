##### What is Domain Name Service (DNS)?
 - It sits in the application layer of the TCP/IP model.
 - Why do we need a DNS server? The reason is the same with having names for contacts that map to certain phone numbers; it makes remembering the contact information for people easier.
 - Every time you would like to connect to a web site, you would have to remember their IP address, i.e. very difficult.
 - This is a problem! So, we use a DNS server to map **Domain Names** to IP addresses.
 - E.g. `google.com` is a domain name.
 - The process of translating a domain name to an IP address is known as **name resolution**.
 - Simplified Process:
	 - User to browser: I want to access "example.com".
	 - Browser to DNS: What is the IP address of this domain name "example.com"?
	 - DNS to browser: Either the IP address is found, e.g. "`104.18.27.120`", or its not found.
	 - Browser to web server: Using the received IP address, asks the server for the requested web site.
	 - Web server to browser: Here is the webpage or other resource requested.
	 - Browser to user: Displays the received data.


##### URL Structure
 - Domain names take the form of a **Uniform Resource Locator (URL)**.
 - A URL is a form of a Uniform Recourse Identifier (URI).
 - Common URL structure: ![[Pasted image 20260408113535.png]]
	 - Everything in square brackets `[]` is optional.
	 - `scheme`
		 - The protocol the browser must to use to request the resource.
		 - E.g. HTTPS or FTP.
	 - `://`
		 - Indicates the end of the `scheme` and the start of the `userinfo` or `host`.
	 - `userinfo`
		 - This is an old and discouraged method to include authentication details in the URL directly.
		 - E.g. `username:password@`
		 - It ends with the `@` symbol.
	 - `host`
		 - Also known as the **authority** or **domain name**.
		 - An example of the domain name is `example.org` and the subdomain is the `www`.
	 - `port`
		 - This is the specific port on the server to access the data.
		 - Usually `80` for HTTP and `443` for HTTPS.
		 - Starts with `:`.
	 - `path`
		 - This shows the specific "route" or path on the web server where the resource is located.
		 - The use to point to specific files, but now they are virtual and point to specific database records.
	 - `query`
		 - Also known as **parameters**.
		 - These provide more information to the web server to filter or modify the request further.
		 - They start with `?` and are separated by `&`.
		 - Each parameter is a key-value pair in the form `key=value`.
	 - `fragment`
		 - Also known as an **anchor**.
		 - It identifies a specific location within the resource itself.
		 - They start with the `#` symbol and are not sent to the web server.
		 - Once the browser receives the web page, it will "look" for an element on the page with the ID provided in the fragment, they automatically scroll to that element.

##### Mapping Hostnames to IP Addresses
 - DNS mappings are not just one-to-one.
 - *Multiple Hostnames to One IP Address*
	 - Shared / virtual hosting
	 - Different hostnames map to the same IP address.
	 - E.g. `example.com`, `example.co.uk`, etc, all map to `200.100.50.10`.
 - *One Hostname to Multiple IP Addresses*
	 - Load balancing - spread the load of requests over multiple servers.
	 - Redundancy - Ensuring no single point of failure.
	 - However, you get data redundancy, where the data may not be consistent across different servers.


##### DNS Hierarchy
 - DNS is the backbone of the internet. ![[Pasted image 20260408114115.png]]
 - `Root`
	 - This is represented by a single dot (`.`).
	 - You don't usually type this in the browser, but it is implicitly there: `google.com.`.
	 - The root servers don't store website data, rather pointers to TLD servers.
	 - You have about 2014 physical root servers spread around the planet.
	 - They are operated by 13 logical server operators.
	 - They are the first contact if a resolver has not cached a domain name.
 - `Top Level Domain (TLD)`
	 - These are the most general parts at the end of a URL, and are categorised into the following: generic and country code.
	 - **Generic TLDs (gTLDs)**
		 - These indicate the purpose of the site.
		 - `.com` for commercial, `.org` for organisation, or `.gov` for government.
	 - **Country-code TLDs (ccTLDs)**
		 - These are reserved for specific countries.
		 - E.g. `.uk` for United Kingdom.
	 - They return a pointer/IP address to the respective lower domain server to contact next.
 - `Lower Domains (Second Level Domains)`
	 - Can be called authoritative name servers.
	 - This is where individuals and businesses register their names.
	 - E.g. `google.com`, `google` is the second level domain.
	 - These must be unique within their specific TLD, i.e. one person can own `example.com`, but another can own `example.org`.
	 - These servers hold the actual resource records, so can return the IP address to the specific domain name, e.g. `example.com`.
 - `Hostname/Sub-domains`
	 - These are created by the owner of the second level domain to organise their own internal resources.
	 - The sub-domains are used to create logical divisions of a site, such as `en` for the English version of Wikipedia (`en.wikipedia.org`).
	 - Hostnames are often used to identify specific services or serves, like `www` for the web server, `mail` for an email server, and so on.
 - When you try to visit a site, your browser analyses the address from *right-to-left*.
	 - It asks the Root where to find the `.org` servers.
	 - It asks the `.org` TLD where to find the Wikipedia servers.
	 - It then asks the `wikipedia.org` servers where to find the `en` sub-domain.
 - This is method makes the internet scalable!
 - The resolver is the server who actually asks the root, TLD, etc servers where to find the IP address of the requested server.

##### DNS Security Problems
 - **Threats**
	 - Forged DNS replies - fake replies.
	 - Cache poisoning - Manipulating the DNS cache on your local machine or on the recursive resolver.
	 - On path tampering - A man-in-the-middle tampers with the DNS reply.
 - **Impact**
	 - Redirects users to a wrong or malicious service
 - **Need**
	 - Authenticity and integrity for DNS data
 - **Solution**
	 - DNSSEC adds a layer of trust on top of DNS (digital signature)
	 - Digital Signature - Says that actually belongs to this "person".


##### DNSSEC
 - Adds authentication (ensure you know the data came from the server you were expecting) and integrity (lets you know if the data has been tampered with during transport)
 - However, it doesn't add confidentiality, i.e. data is still sent as plain text.
