##### What is Branch Prediction?
 - Branch instructions control the flow of a program.
 - When a branch is encountered, the program needs to decide whether to jump to another part of the code or continue with the next immediate instruction.
 - Branches are the low level technical name for if statements.
 - A branch predictor "guesses" the outcome of a branch instruction before it is even evaluated.
 - The aim is to prepare the CPU for the "predicted" instruction after the branch instruction to improve the efficiency of the CPU.


##### Types of Branch Prediction
 - **Static Branch Prediction**
	 - Make a fixed prediction based on a simple rule, without considering runtime behaviour.
	 - Common techniques: always taken, always not taken, backward branch (loop).
	 - The most common static predictor is: "forward not taken, backward taken", meaning that in an if statement (forward), you never "jump", but in a loop "backward", you always "jump".
 - **Dynamic Branch Prediction**
	 - This predictor changes its prediction based on runtime behaviour.
	 - Common predictors: 1-bit predictor, 2-bit predictor, Branch History Table (BHT), and Global Branch Prediction.
 - **Hybrid Branch Prediction**
	 - Combine multiple branch predictors to improve accuracy.
	 - At first, one is chosen as the starting predictor, then as the program progresses, each prediction is compared and switches are made if one predictor is performing better than the other.
	 - E.g. Tournament Predictor.

##### Speculative Execution
 - Branch prediction and speculative execution work hand in hand.
 - The branch predictor simply predicts if the upcoming branch is taken or not taken.
 - Then speculative execution actually acts upon the prediction made by the predictor.
 - I.e. if the branch predictor predicts the upcoming branch is taken, speculative execution will start to fetch and decode and instructions the branch jumps to, before the branch has been evaluated.
 - The *instructions that are fetched during speculative execution* are known as **speculative instructions** as the CPU doesn't actually know if they are correct until it has evaluated the branch.
 - How it works:
	 - The CPU keeps track of the state of registers, memory and other resources (i.e. the **architectural state**), so in the case of incorrect prediction, the system state can be rewound.
	 - Before the branch has been resolved, the speculative results (instructions) are stored in a **Reorder Buffer** before they are either discarded or saved to the architectural state.
	 - If the "speculation" (prediction) is correct, the results (speculation state) is committed to the CPU's *architectural state* (the state "visible" to applications).
	 - However, if the speculation is incorrect (i.e. the prediction was wrong), the speculative instructions are discarded, the pipeline is flushed, and the CPU rolls back to the correct state before the branch.
	 - The CPU the resumes execution down the other path.

##### In-order vs. Out-of-order Execution
 - **In-order**
	 - Instructions are executed in the exact sequence they appear in the program.
	 - Therefore, if one instruction is waiting for a memory read, the entire pipeline must stall, waiting for that instruction to complete.
 - **Out-of-order**
	 - The CPU doesn't have to wait for earlier instructions to complete if later ones are ready for execution.
	 - It can bypass stalled instructions and execute others that have the necessary data and resources.
	 - To retain the correct program logic, the CPU must "commit" (finalise) the executed instruction in-order.
	 - The CPU maintains an instruction pool, where instructions are executed out-of-order. ![[Pasted image 20260409124100.png]]
	 - You use a data flow diagram to identify which instructions depend on the results of previous instructions. This way, the CPU can execute instruction out-of-order, but maintain the program logic: ![[Pasted image 20260409124243.png]]
		 - Here, instruction 2 relies on the result of instruction 1, inst.5 requires inst. 3 and 4, and then inst. 6 requires inst. 2 and 5.
		 - Therefore, the CPU is able to execute instructions 1, 3, and 4, out-of-order.
		 - This reduces the total time to execute instructions 1 through 6, but each individual instruction takes the same amount of time.


##### Principle of Locality
 - **Locality** is a *fundamental concept* in computer architecture that states program tend to access a small, localised portion of memory repeatedly over short periods of time.
 - There are two main types:
	 - **Temporal Locality (Locality in Time)**
		 - If a memory location is accessed, it is likely to be *accessed again soon*.
		 - Example: A loop iterating over an array repeatedly accesses the same variables multiple times.
	 - **Spatial Locality (Locality in Space)**
		 - If a given memory location is access, *nearby locations* are likely to be *accessed soon*.
		 - Example: when accessing elements in an array, adjacent elements are often accessed next.


##### 90-10 Rule of Thumb
 - The idea is that the program spends ~90% of execution time in just ~10% of the code.
 - The exact percentages can change, but the idea is the same.
 - Not all parts of a program are equally important for performance.
 - Key Takeaways:
	 - Focus on optimising the 10% of code, rather than the premature optimisation of the entire program.
	 - Once key bottlenecks are optimised, further improvements may yield minimal gains.
	 - When optimising, focus on the frequent case, as the improvement is higher if the occurrent is frequent, i.e. a loop of some kind.
 - Amdahl's law quantifies common case performance improvement.


##### Amdahl's Law
 - **Amdahl's Law** states that the performance improvements to be gained from using some faster mode of execution (or parallelism) is limited by the fraction of time that the faster mode can be used!
 - I.e., if the faster code is only used 20% of the time, then you can only improve that 20% of time.
 - Formula to calculate the speedup: ![[Pasted image 20260213161125.png]]
 - The overall speedup is comprised of two parts:
	 - **Enhancement fraction**
		 - The fraction of the computation time that can be converted to take advantage of enhancement.
		 - Example: if a task takes 60 seconds, but 20 seconds can benefit from an enhancement, then $\text{Fraction}_\text{Enhanced} = \frac{20}{60}$.
		 - This fraction is always $\leq1$
	 -  **Speedup Fraction**
		 - The improvement gained by the enhanced execution mode.
		 - I.e. how much faster the task would run if the enhanced mode were used for the entire program.
		 - Fraction = time of original over time of enhanced.
		 - Example:
			 - Say a task took 10 seconds, but with enhancement it takes 2 seconds.
			 - Then, $\text{Speedup}_{\text{Enhanced}} = \frac{10}{2} = 5\text{x improvement}$.
			 - This speedup fraction will always be $\geq1$, because you will either improve for stay the same.
 - Overall speed up formula:
$$\text{Speedup}_{\text{Overall}} = \frac{\text{Execution Time}_{\text{old}}}{\text{Execution Time}_{\text{new}}} = \frac{1}{(1-\text{Fraction}_{\text{Enhanced}})+\frac{\text{Fraction}_{\text{Enhanced}}}{\text{Speedup}_{\text{Enhanced}}}}$$
