##### What Memory Hierarchy?
Memory hierarchy is divided into 4 main levels:

| Level                       | 1                   | 2                                  | 3                       | 4                       |
| --------------------------- | ------------------- | ---------------------------------- | ----------------------- | ----------------------- |
| Name                        | Registers           | Cache                              | Primary Storage         | Secondary Storage       |
| Size                        | $<1 \; \text{KB}$   | $32 \; \text{KB} - 8 \; \text{MB}$ | $\leq 512 \; \text{GB}$ | $\geq 1 \; \text{TB}$   |
| Tech                        | CMOS                | CMOS SRAM                          | CMOS DRAM               | SSD, HDD, CD, DVD, Tape |
| Speed ($\text{ns}$)         | $0.15-0.3$          | $0.5-15$                           | $30-200$                | $200-5,000,000$         |
| Bandwidth ($\text{MB/sec}$) | $100,000-1,000,000$ | $10,000-40,000$                    | $5,000-20,000$          | $50-20,000$             |
| Managed by                  | Compiler            | Hardware                           | OS                      | OS/User                 |
| Backed by                   | Cache               | Primary Storage                    | Secondary Storage       | Other secondary storage |


##### Key Cache Concepts
To improve performance, processors use cache to store frequently accessed data. Key metrics include:
 - **Hit Rate**: The fraction of memory accesses that are found in the cache.
 - **Miss Rate**: The fraction of memory accesses not found in cache (1 - hit rate).
 - **Hit Time**: The time required to access a level of memory hierarchy when it is a hit.
 - **Miss Penalty**: The extra time to find some data in a lower level (i.e. RAM or lower) and copy it to a higher level, plus the time to deliver it to the processor.
 - **Block**: The smallest unit of information that can be moved between memory levels.


#### Cache Design
##### Physical Organisation
This is the structure of a memory address: ![[Pasted image 20260409134015.png]]
 - **Block Address**
	 - Indicates the block number in memory we need to access.
	 - It is formed from the higher order bits, i.e. the left most bits.
 - **Block Offset**
	 - This tells you where the specific data is WITHIN the block.
	 - Formed using the remaining bits, right most.
 - Block address if further broken down into index and tag field.
 - **Index or Set Field**
	 - Indicates the specific set (group of blocks) in the cache where the data should be stored.
	 - For a cache with $2^n$ sets, the index field must be $n$ bits long.
 - **Tag Field**
	 - This is a unique identifier for a block.
	 - If multiple blocks from main memory map to the same block in cache, the CPU uses this tag to identify which nearby block is the requested one.
	 - I.e., make sure this is the block you want.


##### Logical Organisation
When organising cache, you must answer several questions:


###### Block Placement
 - *Q*: Where and how can we place a block?
 - **Directly-mapped**
	 - Each memory location maps to exactly one specific cache location.
	 - Formula for placement: $\text{[Block Address] MODULO [Number of blocks in cache]}$.
 - **Fully Associative**
	 - A memory block can be placed anywhere in the cache.
 - **Set-Associative**
	 - A block maps to a specific "set" of locations and can be placed anywhere within that set.
	 - A set is a group of blocks in the cache.
	 - The block is first mapped to a set, then it can be placed anywhere within that set.
	 - If there are $n$ blocks in a set, it is known as $n$-way set associative.
	 - Formula for placement: $\text{[Block Address] MODULO [Number of sets in cache]}$.
 - Using set association, directly mapped is where each set can only have one block, and fully associative is where the set is the size of the cache, i.e. 1 set for all blocks.
 - Relating back to the block address, the index is also called the set.

###### Block Identification
 - *Q*: How do we find a block in the cache?
 - Here is a diagram explaining how to find a block:![[Pasted image 20260409140349.png]]
 - The valid field identifies if this block has been changed somewhere else and needs updating.
 - For directly mapped cache:
	 - The *index* is used to locate the specific block in cache.
	 - The *tag* of the block at the *index* is compared with the tag in the provided block address, if they match, you have found the requested block.
 - For 4-way associative (4 blocks in a set):
	 - The *index* is used to locate the set of 4 blocks.
	 - Then, the *tag* is compared to each of the 4 blocks in the set.
	 - The block with the matching tag has its data sent to the CPU.
 - If a miss occurs, the cache controller must then search the memory one level lower in the hierarchy, i.e. RAM next.


###### Block Replacement
 - *Q*: Which block should be replaced on a miss?
 - When a miss occurs, the cache controller must select a block to be replaced with the desired data.
 - With directly mapped cache, there is no choice since there is only one block that can be replaced, therefore this block is replaced with the desired data.
 - However, for set associative and fully associative:
	 - **Random Replacement**
		 - A block to replace is chosen at random.
	 - **Least Recently Used (LRU)**
		 - To reduce the chance of throwing out a block that may be needed soon, the block that is lest recently used, is the perfect candidate for replacement.
	 - **First In, First Out (FIFO)**
		 - Because LRU can be complicated to calculate, this approximates LRU by determining the oldest block rather than the LRU.
		 - I.e., the first block to enter the cache is the first to be replaced.
		 - Similar to a queue, first in, first out.


###### Write Strategy
 - *Q*: What happens on a write?
 - **Write Through**
	 - The information that is changed is written to the block in the cache and in the lower level memory (i.e. RAM and lower).
	 - This is easier to implement than write back.
	 - It also means that each lower level of the memory hierarchy has the most recent copy of the data, which simplifies data coherency.
 - **Write Back**
	 - The data changed is only written back to the block in the cache.
	 - The block in main memory is only updated when it is replaced for another block.
	 - This means that you can update a block multiple times, with only one costly write to main memory.
	 - This method saves power and is therefore common in embedded systems.
	 - You have a dirty bit feature that is used to reduce the frequency of writing back.
