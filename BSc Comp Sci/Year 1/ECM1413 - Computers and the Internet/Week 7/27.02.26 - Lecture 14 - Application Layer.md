##### What is the Application Layer?
 - The **application layer** provides the services and protocols that applications use to communicate.
 - **Service** = what is being provided
 - **Protocol** = the systems that allows the service to communicate with the client.
 - Protocols used within the application layer: HTTP(S), DHCP, DNS, SMTP, FTP, SSH, and Telnet.


##### Standards - IETF and IEEE
 - **Internet Engineering Task Force (IETF)**
	 - They focus on protocols and software that enable the internet to function.
	 - I.e., the upper layers of the OSI model.
	 - They create RFCs (Request for Comments) that document the standards to follow.
 - **Institute of Electrical and Electronics Engineers (IEEE)**
	 - They are concerned with the hardware and physical networking standards.
	 - I.e., layer 1 and 2 of the OSI model.
	 - They publish journals and create standards like the IEEE 802.11 (Wi-Fi) and IEEE 802.3 (Ethernet).


##### Client-Server Architecture
 - There is **asymmetric roles**, i.e. clients request services and servers provide them.
 - Client-server networks have a **centralised topology**, where the central entity (device) is the server, which must have high capacity and availability.
 - Logically:
	 - A **server** is a process on an end device that *"listens" for requests* for services.
	 - Then, a **client** is a process, also on an end device, that initiates communication by *sending requests* to servers.


##### Peer-to-Peer Architecture
 - Devices within this type of network have symmetric roles, meaning peers (end devices) can both request and provide services/resources.
 - Decentralised topology, direct communication without a central entity
 - These networks have a decentralised topology, meaning peers can communicate directly without a central entity.
 - Every node can request (client) and provide resources (servant).


##### Hypertext Transfer Protocol (HTTP)
 - The web is an *information space* of resources (identified by *URIs*) interconnected by *links*.
 - HTTP is the protocol that facilitates that transfer of these resources between end devices.
 - The client makes a HTTP `GET` request for a particular website.
 - Then, the server responds with a HTTP `Response` containing the webpage.
 - Uniform Resource Identifiers (URIs) are identifiers used to access certain resources on the internet.


##### What is a HTTP Request?
 - This is the "thing" the client, like your web browser, sends to a web server to "request" a web page.
 - You have a request line: e.g. `GET / HTTP/1.1`
	 - `GET` - this is the type of request, i.e. the client "wants" a resource.
	 - `/` - is the path on the web server to which the website is located.
	 - `HTTP/1.1` - indicates the version of the `HTTP` protocol that is being used.
 - You then have the request headers which provide more information about the request: ![[Pasted image 20260407164243.png]]
 -  Other request methods include:
	 - `POST` - add a recourse to the destination
	 - `DELETE` - delete a resource at the destination


##### What is a HTTP Response?
 - This is the type of data the web server sends back to the client after they make a HTTP request.
 - You have the status line, similar to the request line, which states the version of HTTP being used, the status code and message.
 - Similar to requests, you have the response headers, which provide extra information about the response, such as the type, encoding and length of the data being requested, along with the date and time.
 - As alluded to above, then response also contains the data itself, called the body, e.g. the web page requested.
 - The body can be HTML, JSON, images, CSS, JS, etc...


##### Loading a Webpage
 - The browser fetches the main HTML, then requests all referenced recourses, e.g. images, CSS, JS, other HTML, etc.
 - If the browser has cached any responses, it will use them, or it may revalidate them, i.e. re-request them.


##### HTTPS
 - The S stands for secure.
 - *Confidentiality* - It uses encryption, i.e. doesn't send your data as plain text and prevents the "man in the middle" attack!
 - *Integrity* - It protects against tampering, i.e. you will know if the data has change during transport.
 - *Authentication* - It also allows use to authenticate the server we are requesting a service from.
 - HTTP over TLS.
 - HTTP 3 uses QUIC not TCP.
 - You can use APACHE, NGINX, or Microsoft IIS to run a web server.


##### Dynamic Host Control Protocol (DHCP)
 - All devices connected to the network need to have an IP address, hence IP configuration is used.
 - In your home network, your router acts as the DHCP server, providing a private IP address to each connected end device.
 - In an enterprise, they usually have a dedicated server for this task.
 - DHCP uses UDP to transmit the data, i.e. no error correction!
 - Main idea: automate host configuration.


##### Remote Access Protocols (Telnet and SSH)
 - **Telnet** - transfers data as plain text, i.e. no encryption or integrity.
 - **Secure Shell (SSH)** - provides encrypted and integrity protected communication, with server authentication.
 - Example SSH command: `ssh [username]@[hostname or IP address]`.
