##### What is a Stable Sort?
 - A stable sort algorithm is one which can be applied to a sorted list and the order will remain the same.


#### Merge Sort
##### History of Merge Sort
 - Proposed by John von Neumann in 1945, but this is debated.


##### What is Mergesort?
 - How does it link to Quicksort?
	 - Quicksort is about *selecting* an element and dividing the list into two parts and then sorting them separately.
	 - There is also a complementary process called *merging*, where you combine two ordered lists into one.
	 - This link is similar to the relationship between selection and insertion sort!
	 - With Quicksort, the partitioning is the complicated part, similarly, the merging part of mergesort is the equivalently complicated part.
 - *Pros*
	 - Performs the sort in $O(n\log_{2}{n})$ in the worst case, better than Quicksort.
	 - It is *stable* and robust regarding the initial configuration of the list.
	 - Can be adapted to work with linked lists.
 - *Cons*
	 - It may require an additional array of (up to) the same size, meaning this algorithm is not in-place.
	 - However, you can avoid this additional array, but the algorithm becomes significantly more complicated, in which case, you would [[13.03.26 - Lecture 18 - AVL Applications and Heaps|use a heap sort]], which is $O(n\log_{2}{n})$ and is in-place.


##### The Mergesort Idea
 - Idea:
	 1. Calculate the index of the middle of the list, call it `m`.
	 2. Use recursion to deal with the two partitions: `[start,m]` and `[m+1,end]`.
	 3. Merge the two ordered lists into one large list.

#### Radix Sort
##### The Idea of Radix Sort
 - Not all keys are simple integers, they maybe complex strings such as post codes, keys in a phone book, or the key in a library catalogue.
 - Also, in some applications, the analysis of the whole key is not always necessary.
	 - E.g., when looking for a word in a dictionary, how many characters of the word do you need to compare to find this definition?
 - Radix sort tries to build from this assumption to improve efficiency.
 - It does this by decomposing keys into pieces and comparing these individual pieces.
 - A **base** (in the context of radix sort) or **radix** is the *fundamental building block of the keys* to be sorted, e.g. the digits in a number or the characters in a string.
 - Ideas of bases:
	 - We could represent any key in binary and compare the individual bits.
	 - Or, we could consider the individual digits in a number or characters in a string.
 - *Idea*: (for MSD)
	 - Sort the list based on the first base.
	 - This will form "buckets" of keys with the same first base.
	 - Then, each bucket can be sorted using the second base, and so on...
 - *Idea*: (for LSD)
	 - Sort the list based on the least significant digit (right most digit).
	 - Then, sort based upon the next digit, one to the left.
	 - Repeat this for all digits.
 - This idea of buckets is why radix sort is sometimes called bucket sort.

>[!Warning] Radix sort is the idea of splitting up the key, but you use another sorting algorithm, such as merge sort, to sort each radix.


##### Variations of Radix Sort
 - Because radix sort deals with individual radices, there is two ways to compare them:
	 - From left to right
	 - Or, from right to left
 - **Most-Significant-Digit (MSD)** Radix Sorts use the *left* (most significant digit/bit) *to right* idea.
 - **Least-Significant-Digit (LSD)** Radix Sorts use the *right* (least significant digit/bit) *to left* idea.
 - *MSD is preferred* as it *doesn't need to analyse all digits*/radices to sort the list (where LSD does), but is *more complicated to implement*.
 - Why is MSD more complicated?
	 - With LSD, you can simply repeatedly sort each "column" of respective digits, however, with MSD, you must consider the buckets formed when sorting!


##### Representing Radices
 - You represent the radices of keys in different ways depending on the size and variation of them.
	 - For small, uniform keys, you can simply analyse the individual digits in the number (for instance).
	 - However, for larger and non-uniform keys, it is easier to compare them using their binary representation.
	 - You may need to "pad" the binary to make them all the same length!


##### Performance of Radix Sort
 - For sorting $n$ records with $k$ radices, the running time of the radix sort is equivalent to $k \cdot n = O(n)$ since $k$, the size of each key, is constant.
 - Obviously, the performance depends on the sorting algorithm uses to sort each radix.
 - Therefore, the sorting algorithm chosen must be $O(n)$ to achieve $k \cdot n$ performance.
 - However, what sorting algorithms are $O(n)$?
	 - The $n\log_{2}{n}$ limit is only for comparison based algorithms.
	 - Since the values are within a predefined range, $O(n)$ algorithms like the [[#Counting Sort|counting sort]] can be used!
 - For large values of $n$, the running time of radix sort is comparable to $O(n\log_{2}{n})$.
	 - For instance, if we represented our keys in binary and had 1 million 32-bit keys, then $k$ and $\log_{2}{n}$ are both about 32.
	 - Therefore, $k \cdot n$ is comparable to $n \log_{2}{n}$.


#### Counting Sort
##### The Counting Sort Idea
  - Counting sort takes advantage of knowing the range of values it is sorting to improve its efficiency.
  - Therefore, it is a good candidate for radix sort.
  - Idea:
	  - For each element $x$ in the list to be sorted, count how many other elements are less than $x$.
	  - This tells you where element $x$ will be located in the sorted list.
	  - E.g. if there are $20$ elements less than $x$, then the position of $x$ in the sorted list is $21$.


##### Implementing Counting Sort
 - Process:
	 1. You want to create an array, where each index is the value in the original array, and the corresponding value in this new array is the number of elements less than or equal to it. ![[Pasted image 20260402153916.png]]
		 - This looks a bit cryptic, but `C[A[j]] = C[A[j]] + 1` simply generates an array where each value is the number of occurrences of the index "value" in the original array.
		 - E.g., `C[0]` states that there is `1` occurrence of the value `0` in the original array.
		 - Then, `C[i] = C[i] + C[i-1]` simply states for each value in the original array (index in the new array), how many elements there are less than or equal to it.
		 - E.g., new `C[1]` states that there are `3` elements less than or equal to `1`. We can see that this is true, within the original array you have `0, 1, 1`, 3 values!
	 2. Now, starting from the end of the original array, use the value as the index for the array `C` (in this example), where the value held at this index in `C` is the index to place the original value at in the new sorted array `B`. This is what `B[C[A[j]]] = A[j]` does.
	 3. Then, you must decrement the value held in `C`, so the next duplicate value is placed correctly. This is what `C[A[j]] = C[A[j]] - 1` does.
	 4. Repeat steps 2 and 3, until the pointer `j` reaches -1, i.e., you have finished analysing the original array.

##### Analysis of Counting Sort
 - The first loop iterates $k$ times.
 - The second, $n - 1$ times.
 - The third, $k - 2$ times.
 - And the fourth, $n - 1$ times.
 - Therefore, we have a time complexity consisting of:
$$\text{Number of iterations} = k + (n-1) + (k-2) + (n-1) = 2k + 2n - 4 \implies O(k + n)$$
 - Counting sort beats $O(n\log_{2}{n})$ as it is *not a comparison based sort*.
 - Counting sort has a space complexity of $O(2n + k) = O(n)$, where $2n$ is for the original and sorted lists, and $k$ is for the counting list.
 - Therefore, counting sort is not in-place.
 - Counting sort *is stable!!!*
